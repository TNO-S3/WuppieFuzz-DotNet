# WuppieFuzz .NET Coverage Agent

This directory contains the coverage agent for collecting code coverage from .NET targets during fuzzing with WuppieFuzz. The agent wraps Microsoft's `dotnet-coverage` tool to provide on-demand coverage snapshots with reset support, which is essential for iterative fuzzing guidance.

## Overview

The agent works by:

1. Starting a `dotnet-coverage collect --server-mode` process in the background
2. Exposing an HTTP API that the WuppieFuzz fuzzer can query for coverage
3. Converting coverage data from binary format to Cobertura XML
4. Optionally generating and returning an HTML coverage report as a ZIP archive

The target .NET application communicates with `dotnet-coverage` via named pipes (IPC), so the fuzzer doesn't need to modify application code.

## Prerequisites

### .NET SDK
- .NET 8.0 or later (to build and run the agent)

### dotnet-coverage tool
Install the global tool:
```bash
dotnet tool install --global dotnet-coverage
```

Verify installation:
```bash
dotnet-coverage --version
```

### Optional: Report generation
For HTML coverage reports, install `reportgenerator`:
```bash
dotnet tool install --global dotnet-reportgenerator-globaltool
```

If `reportgenerator` is not installed, the agent will still return coverage data in XML format.

## Building the Agent

```bash
cd coverage_agents/dotnet/
dotnet build
```

Or build from the WuppieFuzz root:
```bash
dotnet build coverage_agents/dotnet/
```

## Running the Agent

The agent runs as a background HTTP server. Start it with:

```bash
dotnet run --project coverage_agents/dotnet/ -- --port 6302 --session-id wuppiefuzz
```

### Command-line options:

- `--port <PORT>` (default: 6302)  
  The port on which the HTTP server listens.

- `--session-id <ID>` (default: "wuppiefuzz")  
  Session identifier for the `dotnet-coverage` server. Must be alphanumeric (hyphens and underscores allowed).

### Example:
```bash
dotnet run --project coverage_agents/dotnet/ -- --port 6302 --session-id myapp-fuzzing
```

Once started, the agent logs its configuration and waits for requests:
```
Starting dotnet-coverage collect --server-mode --session-id wuppiefuzz
WuppieFuzz .NET coverage agent listening on port 6302
Session ID : wuppiefuzz
Run target : dotnet-coverage connect wuppiefuzz <your-app-command>
```

## Instrumenting the Target Application

The target .NET application must be connected to the same `dotnet-coverage` session. Start it (in a separate terminal) using:

```bash
dotnet-coverage connect <SESSION_ID> dotnet run --project <YourApp.csproj>
```

Where `<SESSION_ID>` matches the session ID used when starting the agent.

### Example:
```bash
# Terminal 1: Start the coverage agent
dotnet run --project coverage_agents/dotnet/

# Terminal 2: Start the target app
dotnet-coverage connect wuppiefuzz dotnet run --project MyApi/
```

**Important:** The target application must remain running while the fuzzer is active. The agent and target communicate over named pipes; if the target crashes or exits, coverage collection stops.

## Configuring WuppieFuzz

In your WuppieFuzz configuration (YAML file or CLI), specify the dotnet coverage format:

### CLI:
```bash
wuppiefuzz fuzz \
  --coverage-format cobertura \
  --coverage-host 127.0.0.1:6302 \
  api_spec.yaml
```

### YAML configuration file:
```yaml
coverage_format: dotnet
coverage_host: "127.0.0.1:6302"
report: true
```

### Optional: Namespace filtering

To only collect coverage from specific namespaces or file paths, use the `dotnet-namespace-filter` option. This filters based on the source file path containing the filter string.

#### CLI:
```bash
wuppiefuzz fuzz \
  --coverage-format cobertura \
  --coverage-host 127.0.0.1:6302 \
  --dotnet-namespace-filter "MyApp.Controllers" \
  api_spec.yaml
```

#### YAML:
```yaml
coverage_format: dotnet
coverage_host: "127.0.0.1:6302"
dotnet_namespace_filter: "MyApp.Controllers"
```

This will only track coverage from classes whose file path contains `"MyApp.Controllers"`.

## HTTP API Endpoints

The agent exposes the following endpoints:

### GET /health
Returns `200 OK`. Use this to check if the agent is running.

```bash
curl http://localhost:6302/health
```

### GET /coverage
Returns the current coverage as Cobertura XML.

```bash
curl http://localhost:6302/coverage > coverage.xml
```

Query parameters:
- `reset=true` — Take a snapshot and reset the coverage counters in the target application

```bash
curl "http://localhost:6302/coverage?reset=true" > coverage.xml
```

### GET /report
Generates an HTML coverage report from accumulated coverage and returns it as a ZIP archive.

```bash
curl http://localhost:6302/report -o report.zip
unzip report.zip -d ./coverage_report
```

The ZIP contains an HTML report generated by `reportgenerator` (if installed). If `reportgenerator` is not available, the agent returns the accumulated Cobertura XML instead.

### GET /shutdown
Gracefully shuts down the agent and the `dotnet-coverage` server.

```bash
curl http://localhost:6302/shutdown
```

## Architecture

```
WuppieFuzz (Rust)
    ↓ HTTP GET /coverage[?reset=true]
    ↓
Coverage Agent (.NET)
    ↓ named-pipe IPC
    ↓
dotnet-coverage collect --server-mode
    ↓ CLR profiling API
    ↓
Target .NET Application
    (runs via: dotnet-coverage connect <id> dotnet run)
```

The agent maintains a persistent named-pipe connection to `dotnet-coverage`, so each coverage query is low-latency (typically 10–50 ms) without spawning subprocesses per iteration.

## Troubleshooting

### "dotnet-coverage: command not found"
Install the global tool:
```bash
dotnet tool install --global dotnet-coverage
```

Ensure your PATH includes the .NET tools directory (usually `~/.dotnet/tools`).

### Agent crashes or fails to start
- Check that the `dotnet-coverage` server started successfully (port may already be in use)
- Verify the session ID is alphanumeric
- Check log output for IPC errors

### No coverage data returned
- Verify the target app is running and connected to the same session ID
- Test the target app manually to confirm it loads and serves requests
- Check that the source files are being instrumented (look for slow startup)

### "dotnet-coverage snapshot" fails with timeout
- The target app may be unresponsive or hung
- Restart both the agent and target app
- On Linux, ensure named-pipe permissions are correct

### reportgenerator not found (fallback to XML)
- Install reportgenerator: `dotnet tool install --global dotnet-reportgenerator-globaltool`
- Or accept XML output from the agent (still useful for coverage tracking)

## Platform Support

- **Windows**: Fully supported (named-pipe IPC via `\\.\pipe\dotnet-coverage-<id>`)
- **Linux**: Supported with dynamic instrumentation (`dotnet-coverage` v17+, .NET 8+, x64 architecture)
- **macOS**: Supported with dynamic instrumentation (.NET 8+, x64 architecture)

Static instrumentation (via `dotnet-coverage instrument` command) is supported on all platforms but requires pre-processing the target binary before running the agent.

## Example Workflow

### Terminal 1: Start the coverage agent
```bash
cd /path/to/wuppiefuzz/coverage_agents/dotnet
dotnet run -- --port 6302 --session-id fuzzing-session
```

### Terminal 2: Start the target API
```bash
cd /path/to/your/api
dotnet-coverage connect fuzzing-session dotnet run
```

### Terminal 3: Run the fuzzer
```bash
cd /path/to/wuppiefuzz
wuppiefuzz fuzz \
  --coverage-format cobertura \
  --coverage-host 127.0.0.1:6302 \
  --report=true \
  --timeout 3600 \
  api_spec.yaml
```

After fuzzing completes (or at any time), retrieve the coverage report:
```bash
curl http://127.0.0.1:6302/report -o report.zip
unzip report.zip -d ./coverage_results
open ./coverage_results/index.html  # macOS/Linux
```
